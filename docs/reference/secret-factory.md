# Secret Factory Reference

The Secret Factory is the Bears plugin-local write-only path for values that can be generated by the agent and written to Infisical without exposing the value.

## Contract

- Catalog: `assets/catalog/secret-factory.v1.json`.
- Validator and writer: `scripts/secret_factory.py`.
- Skill: `skills/secret-factory/SKILL.md`.
- Primary role: `bears-secret-factory-engineer`.
- Supporting reviewer: `bears-platform-security-reviewer`.

Mandatory refusal classes are:

- `provider_issued_api_key`
- `oauth_client_secret`
- `ssh_private_key`
- `tls_private_key`
- `payment_credential`
- `wallet_private_key`

## Write-only rule

The generated value resides only in in-memory script state. The script posts it to the Infisical API v4 create endpoint and discards the response body. Output contains only status, key name, generator kind, Infisical path, and provider handoff metadata.

The script must not read Infisical values, echo values, store values on disk, pass values through command-line arguments, commit values, log values, or include values in tests or docs.

## Request file

A request file is JSON with these fields:

- `secret_name`: uppercase Infisical key name.
- `kind`: one allowed generator kind or one refusal class.
- `secret_path`: optional Infisical path.
- `bytes`: optional bound for byte-based generators.
- `length`: optional bound for password generation.

The request file must contain only keys from `assets/catalog/secret-factory.v1.json:request_schema` and must not contain `secret_value`, `token`, `credential`, or `private_key`.

The runtime parser and catalog request schema must stay in parity:

- required fields: `secret_name`, `kind`
- optional fields: `secret_path`, `bytes`, `length`
- mandatory forbidden fields: `secret_value`, `token`, `credential`, `credentials`, `private_key`

Variant value-bearing request keys are rejected across camelCase, kebab-case, snake_case, and nested structures.

## Local generation

Allowed kinds are `random_base64url`, `random_hex`, and `random_password`.

## Provider handoff

Provider-issued and custody-bearing values are refused. The handoff packet names the requested key, requested kind, handoff owner, target project presence, target environment, target path, and reason. It never contains value material.

## Names-only handoff readiness

The catalog contains `names_only_handoff_refs` for provider-owned Infisical refs that the active backend lanes need by key name and path only.

Run:

- `python3 scripts/secret_factory.py handoff-readiness`
- `python3 scripts/secret_factory.py handoff-readiness --provider serverspace`
- `python3 scripts/secret_factory.py handoff-readiness --provider gitlab`

The command reports represented refs, sets `live_infisical_existence_checked` to `false`, and marks `infisical_readback` as `forbidden`. It does not call Infisical read or list endpoints and does not prove live secret existence. Operators or provider owners must create or confirm the refs in Infisical without returning values to the agent.

## Documented-unconfirmed ref drift

`documented_unconfirmed` refs are documented mappings only. Infisical MCP `get_secret` may return folder 404 for a stale path, MCP `list_secrets` may lack a names-only or exclude-values mode, and the local CLI may have no valid login session.

Do not treat a `documented_unconfirmed` path as live-confirmed. Do not invent a replacement path.

Before T085 or T109 for any future `documented_unconfirmed` GitLab ref, one of these confirmations is required:

1. Operator-confirmed exact project, environment, path, and key name.
2. Safe metadata-only folder and key-name checks through `list_folders` then `list_secret_names`.

Do not call `list_secrets` for this confirmation unless the MCP surface has a names-only or exclude-values mode.

## Provider API token routing

When a task needs a provider API token and a matching catalog ref exists, agents must follow this order. For `documented_unconfirmed` refs, complete the drift confirmation rule above first. For `operator_confirmed_live_ref` refs, follow the exact catalog route and still resolve the secret at task time.

1. Resolve the secret by exact `secret_name` and `secret_path` through Infisical MCP.
2. Keep the value only in in-memory script state for the immediate provider API request.
3. Print only redacted API metadata, HTTP status, resource IDs, and non-secret names.

Agents must not treat chat-pasted token text, shell history, `.env` files, raw config files, or provider CLI config as the source of truth for provider API work.

For ServerSpace Public API work, the required ref is:

- project: Bears Workspace
- environment: `dev`
- path: `/srv/bears`
- key: `SERVERSPACE_API_KEY`

`S2CTL_APIKEY` refs are legacy CLI lane refs and must not be used for ServerSpace Public API calls.

## Infisical contract

The writer sends the Infisical API v4 create-secret request shape with `projectId`, `environment`, `secretValue`, `secretPath`, and shared type. Required runtime environment variables are `INFISICAL_TOKEN`, `INFISICAL_PROJECT_ID`, and `INFISICAL_ENVIRONMENT`. Optional variables are `INFISICAL_API_URL` and `INFISICAL_SECRET_PATH`.

If `INFISICAL_API_URL` is set, it must be HTTPS and match the catalog network allowlist. Foreign hosts must be rejected before write.

The catalog `infisical_network_policy` is a validation contract. It pins HTTPS, the allowed Infisical host, and redirect-host rejection before any write-only create request can send generated value material.

`scripts/validate_overlay.py` owns repo-local static safety helpers for Secret Factory adjacent governance:

- field-complete plugin manifest forbidden-text checks for user-visible fields;
- raw endpoint, URL, IP, and CIDR literal scan helpers with redacted excerpts;
- secret-discovery path classification for safe wrappers, safe contracts, secret ref names, and forbidden credential file paths;
- inventory-output URI redaction before closeout text;
- live-tool mention classification before PR safety scans fail.

T110 static-literal policy is PR-added-lines first. Pre-existing repository findings require a baseline record or a separate cleanup issue before repo-wide mutation starts. The required PR gate is `validate_overlay_static_safety_pr_gate`; it reports `raw_endpoint_literal` and `raw_ip_or_cidr_literal` counts only, covers `secret_factory_governance` and `infrastructure_network_governance` planes, and emits compact action JSON for the `line_only_redacted_issue` flow.
